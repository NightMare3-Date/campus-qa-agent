## 1. 概述

RAG (Retrieval-Augmented Generation) 的核心步骤之一是将原始文档切割成合适的切片 (chunks)。
切片的质量直接影响检索召回率和最终回答质量。本文档系统记录本项目支持的所有切割策略、
其适用场景、参数配置及实现方式。

### 当前项目环境

| 项目 | 说明 |
|------|------|
| Embedding 模型 | BAAI/bge-small-zh-v1.5 (向量维度 512) |
| 向量库 | Chroma (Cosine Similarity) |
| 加载方式 | DirectoryLoader(glob="**/*.md", loader_cls=TextLoader) |
| 源文档格式 | Markdown (UTF-8) |

## 3. 策略一：递归字符切割 (RecursiveCharacterTextSplitter)

### 原理

按优先级顺序使用分隔符列表，优先在更自然的位置切割：
段落边界 → 行尾 → 句号 → 感叹号 → 问号 → 逗号 → 空格 → 字符级回退。

### 参数配置

```python
RecursiveCharacterTextSplitter(
    chunk_size=500,          # 目标切片字符数
    chunk_overlap=50,        # 切片重叠字符数 (10%)
    separators=[
        "\n\n",              # 段落分割 (最高优先级)
        "\n",                # 行分割
        "。", "！", "？",    # 中文句子边界
        "，",                # 逗号（短语边界）
        " ",                 # 英文空格
        "",                  # 字符级回退
    ]
)
```

### 适用场景

- **通用知识文档**：结构不固定的 Markdown 文件
- **快速原型**：开发早期阶段，需要快速验证 RAG 效果
- **短文本**：每条文档长度在几百到几千字

### 优缺点

| 优点 | 缺点 |
|------|------|
| 实现简单，零依赖 | 可能切断完整语义段落 |
| 计算开销小 | 对长文档效果有限 |
| 参数直观易调 | 无法感知文档结构 |

## 4. 策略二：语义切割 (Semantic Chunking)

### 原理

利用 Embedding 模型的语义相似度来检测文本中的"语义边界"。
具体做法：将文本细粒度切分（按句子），然后计算相邻句子之间的向量相似度，
当相似度低于阈值时，判定为语义边界并进行切分。

### 算法流程

```
输入: 原始文本
  │
  ▼
1. 句子分割 (按 。！？\n 切分)
  │
  ▼
2. 对每个句子计算 Embedding 向量
  │
  ▼
3. 计算相邻句子间的 Cosine 相似度
  │
  ▼
4. 相似度 < 阈值 (默认 0.45) → 语义边界
  │
  ▼
5. 合并连续低相似度区域，形成最终 Chunk
  │
  ▼
输出: Document chunks
```

### 参数配置

```python
SemanticChunker(
    embeddings_model=HuggingFaceEmbeddings(...),  # bge-small-zh-v1.5
    breakpoint_threshold_type="percentile",        # 阈值类型
    breakpoint_threshold_amount=0.45,              # 百分位阈值
)
```

### 适用场景

- **长篇文章**：多主题、多段落的连续性文本
- **教材章节**：自然段落间有明显主题转换
- **FAQ 集合**：每个问答对天然构成独立语义单元

### 优缺点

| 优点 | 缺点 |
|------|------|
| 保留完整语义边界 | 计算成本高 (多次 Embedding) |
| 避免跨主题污染 | 阈值需要调参 |
| 检索质量更高 | 对短文本效果不明显 |

## 5. 策略三：分层切割 (Hierarchical Chunking)

### 原理

维护两层切片索引：
- **父切片 (Parent Chunk)**: 较大的完整段落 (800-1200 字符)，保留上下文信息
- **子切片 (Child Chunk)**: 较小的详细片段 (200-400 字符)，提高检索精度

检索时：在子切片中搜索 → 找到后返回对应的父切片作为 LLM 上下文。
这种方式兼顾了"检索精度"和"上下文完整性"。

### 算法流程

```
输入: 原始文档
  │
  ▼
1. 按 Markdown 标题/段落分割成粗粒度段落 (Sections)
  │
  ▼
2. 对每个 Section:
     ├─ 作为父切片 (Parent Chunk) → 存入向量库
     │   metadata: {level: "parent", section_index: i}
     │
     └─ 递归切分成子切片 (Child Chunks)
         metadata: {level: "child", parent_index: i, chunk_index: j}
  │
  ▼
3. 子切片向量检索 → 找到父切片 ID → 返回父切片全文
  │
  ▼
输出: 父切片级别的上下文 (尺寸更大的相关段落)
```

### 参数配置

```python
parent_splitter = RecursiveCharacterTextSplitter(
    chunk_size=1000, chunk_overlap=100,
    separators=["\n\n", "\n", "。", "！", "？", "，", " ", ""]
)
child_splitter = RecursiveCharacterTextSplitter(
    chunk_size=300, chunk_overlap=30,
    separators=["\n", "。", "！", "？", "，", " ", ""]
)
```

### 适用场景

- **规章制度**：条文需要上下文才能准确理解
- **技术文档**：API 说明、配置手册
- **法律法规**：条款需结合章节背景解读

### 优缺点

| 优点 | 缺点 |
|------|------|
| 检索精度高 + 上下文完整 | 需要两倍存储空间 |
| 减少"只见树木不见林" | 检索流程略复杂 |
| 支持跨段落关联 | 对扁平文档收益有限 |

## 6. 策略四：混合检索 (Hybrid Search)

### 原理

同时使用向量相似度搜索和关键词 BM25 搜索，将两者结果加权融合，
弥补纯语义检索对精确关键词匹配的不足。

### 检索融合公式

```
score = α × vector_score + (1-α) × keyword_score
```

其中 α 为语义权重，默认 0.7；keyword_score 由 BM25 算法计算。

### 适用场景

- **专有名词密集**：人名、地名、课程编号等
- **中英文混排**：代码片段、缩略语
- **生产环境**：对检索鲁棒性要求高的场景

### 优缺点

| 优点 | 缺点 |
|------|------|
| 语义 + 关键词双重保障 | 需要维护倒排索引 |
| 对专有名词检索稳定 | 融合权重需要调优 |
| 检索召回率更高 | 有一定计算开销 |

## 7. 策略选择指南

| 你的文档类型 | 推荐策略 | 理由 |
|-------------|----------|------|
| 短 FAQ / 词条 (100-300 字/条) | 递归字符切割 | 简单高效，信息密度高 |
| 长文章 / 教材章节 (数千字) | 语义切割 | 保留主题边界，避免混入不相关内容 |
| 结构化文档 (含标题层级) | 分层切割 | 标题即天然段落边界，父子切片互补 |
| 规章制度 / 法律条文 | 分层切割 | 上下文依赖强，子切片检索 + 父切片回答 |
| 代码 / 技术参考 | 混合检索 | 函数名、类名等精确匹配需求高 |
| 生产环境（不确定文档类型） | 分层切割 + 混合检索 | 覆盖最全面，鲁棒性最强 |

## 8. 参数调优建议

### chunk_size (切片大小)

| 场景 | 推荐值 | 说明 |
|------|--------|------|
| 短文本问答 | 200-400 | 每个切片包含1-2个问答对 |
| 一般知识 | 500-800 | 默认值，适用性最广 |
| 长上下文推理 | 800-1200 | 保留更多上下文，适合复杂推理 |
| 文档摘要 | 1500-2000 | 大切片，适合需要全文理解的场景 |

### chunk_overlap (重叠比例)

| 场景 | 推荐值 | 说明 |
|------|--------|------|
| 非结构化文本 | 10-15% | 默认值，防止关键句被切断 |
| 密集信息文本 | 15-20% | 信息密度高，提高召回率 |
| 结构清晰的文档 | 5-10% | 结构已有天然分隔，减少冗余 |

### Top-K (检索数量)

| 场景 | 推荐值 | 说明 |
|------|--------|------|
| 简单事实查询 | 3 | LLM 上下文窗口小，避免噪声 |
| 综合问答 | 5 | 需要多角度信息 |
| 文档分析 | 8-10 | 大上下文窗口，充分获取信息 |

## 10. 质量评估指标

| 指标 | 说明 | 目标值 |
|------|------|--------|
| **检索命中率** | 前 k 个结果中包含正确答案的比例 | ≥ 85% |
| **语义连贯性** | 切片内句子是否属于同一主题 (人工评定) | ≥ 90% |
| **信息完整度** | 切片是否截断关键信息 (人工抽样) | ≤ 5% 截断率 |
| **响应时间** | 切片 + 向量化耗时 (1000 字文档) | ≤ 1.5s |
| **存储效率** | 切片总数 vs 原始文档字数比 | 1.0~1.3x |

## 11. 后续优化方向

1. **Agentic Chunking**: 用 LLM 自主判断文本边界，动态决定切割位置
2. **多模态切片**: 支持表格、图片、代码块的结构化保留
3. **增量索引**: 仅对新/修改的文档重新切片，避免全量重建
4. **查询重写**: 对用户问题进行改写/扩展后再检索 (HyDE)
5. **Reranker**: 在检索后引入 Cross-encoder 重排序，提升 top-k 质量
6. **缓存层**: 对高频查询结果做缓存，减少重复计算
